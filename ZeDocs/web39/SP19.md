## 1. Périmètre et objectifs

- Installer une instance **Odoo 19** connectée à :
  - **Dorevia Vault** (vaulting des factures et paiements)
  - **Linky** (cockpit, réconciliation bancaire)
- Conserver les instances **Odoo 18** existantes sans modification.

---

## 2. Ce qui reste inchangé (protocoles)

| Flux | Protocole | Impact Odoo 19 |
|------|-----------|----------------|
| **Linky → Odoo** | `GET /dorevia/vault/linky_bank_reconciliation` (API HTTP) | Aucun — API indépendante de la version |
| **Odoo → Vault** | DVIG ingest, proof (HTTP/REST) | Aucun |
| **Réseau** | VAULT_URL, DVIG_URL, tenant, etc. | Aucun |

## 3. Ce qui doit être adapté

### 3.1 Image et configuration

| Élément | Actuel (Odoo 18) | Odoo 19 |
|--------|-------------------|----------|
| Image Docker | `odoo:18.0-20250819` | `odoo:19.0` ou `odoo:19` |
| Dockerfile | `FROM odoo:18.0-20250819` | `FROM odoo:19.0` |
| Manifest tenant | `"odoo": "odoo:18.0-20250819"` | `"odoo": "odoo:19.0-20260205"` (tag épinglé) |

### 3.2 Addons OCA (versions 19.0)

À vérifier sur [OCA](https://github.com/OCA) :

- **account_reconcile_oca** (actuellement 18.0.1.0.16) → branche 19.0
- **account_statement_base**
- **account_reconcile_model_oca**
- **base_sparse_field**
- **queue_job** (pour vaulting temps réel en prod)

### 3.3 Addons Dorevia

- **dorevia_vault_connector** : dépend de `account_reconcile_oca`, hooks sur `account.move`, `account.payment`, `account.bank.statement.line`
- **dorevia_session_guard** : dépend de `ir.http`, `res.users`

## 4. Stratégie des branches (sans perturber Odoo 18)

### 4.1 Principe

| Environnement | Image Odoo | Addons (branche) |
|---------------|------------|-------------------|
| **Existant** (laplatine2026, sarl-la-platine, etc.) | `odoo:18.0-20250819` | `dorevia_vault_connector` branche **18.0** |
| **Nouveau** (instance Odoo 19) | `odoo:19.0` | `dorevia_vault_connector` branche **19.0** |

### 4.2 À faire

1. Créer une branche Git `19.0` (ou `odoo-19`) pour le connecteur
2. Adapter le code uniquement dans cette branche
3. Laisser la branche `18.0` / `main` inchangée pour les instances Odoo 18
4. Déployer Odoo 19 avec un chemin d'addons dédié (ex. `custom-addons-19/`) pointant vers la branche 19.0

### 4.3 À éviter

- Modifier les fichiers utilisés par Odoo 18 pour y ajouter du code Odoo 19
- Remplacer la branche 18.0 par la 19.0 dans les déploiements actuels

---

## 5. Points de vigilance techniques

| Risque | Détail |
|--------|--------|
| **reconcile_bank_line() / unreconcile_bank_line()** | Utilisées par le connecteur Vault pour émettre les événements de lettrage. Si OCA modifie la signature ou le comportement en 19.0, adapter `account_bank_statement_line.py` |
| **account.move et account.payment** | Hooks (`write()`, `action_post()`) et champs peuvent changer entre 18 et 19. En 19, `account.payment` a légèrement changé — `action_post()` peut se déclencher à un moment différent. Surveiller `account_move_line_ids` (souvent là que ça casse). |
| **Python** | Odoo 19 peut imposer une version Python plus récente (ex. 3.12+) — adapter le Dockerfile si besoin |

## 6. Tenant et DNS

### 6.1 Création du tenant

Structure minimale pour un tenant Odoo 19 :

```bash
# Exemple : tenant o19
mkdir -p tenants/o19/{state,secrets,platform,apps/odoo/{lab,stinger,prod}}
```

**Manifest** `tenants/o19/state/manifest.json` :

```json
{
  "version": "1.0",
  "tenant_id": "o19",
  "tenant_mode": "lab",
  "erp": "odoo",
  "created_at": "2026-03-04T00:00:00Z",
  "universes": ["odoo", "ui"],
  "environments": ["lab"],
  "domain_mode": "saas",
  "units": {
    "odoo": ["odoo", "postgres"],
    "ui": ["linky"]
  },
  "images": {
    "odoo": "odoo:19.0-20260205",
    "postgres": "postgres:16",
    "linky": "dorevia/linky:governance-2026-03-03"
  },
  "linky_vault_url": "http://vault-core-stinger:8080",
  "linky_company_display_names": {"odoo:1": "Ma Société"},
  "linky_dvig_url": "http://dvig-core-stinger:8080",
  "linky_dvig_internal_token": "dvig_internal_core-stinger_stinger"
}
```

- `tenant_mode` : mode du tenant (lab, stinger, prod) — Linky en a besoin
- `erp` : ERP producteur des événements (odoo, futur : ERPNext, Pennylane, Shopify)
- Image Odoo : tag **épinglé** pour éviter upgrade silencieux

### 6.2 URLs générées (pattern SaaS)

| Service | URL (lab) | URL (stinger) |
|---------|-----------|---------------|
| **Odoo** | `odoo.lab.o19.doreviateam.com` | `odoo.stinger.o19.doreviateam.com` |
| **Linky** | `ui.lab.o19.doreviateam.com` | `ui.stinger.o19.doreviateam.com` |
| **Vault** | `vault.o19.doreviateam.com` | idem |
| **DVIG** | `dvig.o19.doreviateam.com` | idem |

### 6.3 DNS à configurer

Les hostnames doivent résoudre vers l’IP du serveur où tourne la gateway (Caddy) :

| Hostname | Type | Cible |
|----------|------|-------|
| `odoo.lab.o19.doreviateam.com` | A ou CNAME | IP du serveur / load balancer |
| `ui.lab.o19.doreviateam.com` | A ou CNAME | IP du serveur |
| `vault.o19.doreviateam.com` | A ou CNAME | IP du serveur |
| `dvig.o19.doreviateam.com` | A ou CNAME | IP du serveur |

**Vérification** : `dig +short odoo.lab.o19.doreviateam.com`

### 6.4 Commandes de déploiement

```bash
# 1. Valider le tenant
bin/dorevia.sh validate o19

# 2. Générer les artefacts (compose, Caddyfile)
bin/dorevia.sh render o19 --env lab

# 3. Déployer (platform + apps)
bin/dorevia.sh apply o19 --env lab --auto-gateway

# 4. Si gateway aggregate manuel (déjà fait par --auto-gateway)
bin/dorevia.sh gateway aggregate --reload
```

### 6.5 Réutilisation d’un tenant existant

Pour utiliser Odoo 19 sur un tenant déjà en place (ex. `laplatine2026`) :

- Modifier le manifest : `"odoo": "odoo:19.0"` dans `images`
- Créer un chemin d’addons dédié pour les versions 19.0
- Re-render et redéployer
