# ADR-0010 : Architecture Phase 3 — Snapshots Évolution (Trésorerie, Encours, BFR)

**Statut** : ✅ **Accepté**  
**Date** : 2026-03-14  
**Décideurs** : Équipe Dorevia (produit + technique)  
**Contexte** : Bloc Évolution 3 cards Linky — séries temporelles Trésorerie, Encours, BFR (Phase 3 data)

## 1. Contexte

### 1.1 Problème

Les cards Trésorerie, Encours et BFR ont un bloc Évolution en état **empty** tant qu’aucune série temporelle fiable n’est exposée. La Phase 3 du plan data prévoit une **historisation** (snapshots) puis des endpoints de série. Pour implémenter E2, E3 et E4 (tables, jobs, endpoints, front), l’équipe devait trancher :

- **Où** stocker les snapshots (tables) ?
- **Qui** héberge et exécute les jobs de snapshotting ?
- **Comment** les séries sont exposées (Vault vs API Linky, contrat) ?
- **Quelle** convention de date (dernier jour calendaire vs ouvré) ?
- **Quelle** granularité pour la v1 ?
- **Source des données** : acter la règle « Vault = source exclusive ».

### 1.2 Périmètre

- **Snapshots concernés :** Trésorerie (position), Encours (AR), BFR (AR + AP).
- **Usage :** alimentation des blocs Évolution Linky (format aligné spec consolidée §11).
- **Règle d’architecture (spec §2.0) :** toutes les données métier affichées par Linky proviennent **exclusivement du Vault** (ou de tables alimentées par des jobs qui s’appuient sur le Vault). Aucune autre source pour les montants ou agrégats.

---

## 2. Décisions

### 2.1 Emplacement des tables

**Décision :** les tables de snapshots sont stockées dans la **BDD du service Vault**.

**Motif :**
- cohérence avec la règle **Vault = source exclusive** ;
- pas de duplication de responsabilité métier dans Linky ;
- simplification du tracé de données et des futures migrations.

**Convention retenue :**
- schéma dédié recommandé : **`snapshots`**
- tables v1 :
  - `snapshots.treasury_position_snapshots`
  - `snapshots.ar_totals_snapshots`
  - `snapshots.ap_totals_snapshots`

**Remarque :**
- **Encours** lit la série issue de `ar_totals_snapshots`
- **BFR** est calculé à partir de `ar_totals_snapshots` + `ap_totals_snapshots`
- aucune table BFR dédiée n’est créée en v1, sauf contrainte technique ultérieure explicitement justifiée

---

### 2.2 Responsabilité des jobs

**Décision :**
- la **logique de snapshotting** est implémentée dans le **service Vault**
- le **déclenchement** est assuré par un **orchestrateur externe / cron d’infrastructure**
- la **timezone de référence** est **Europe/Paris**

**Fréquence v1 :**
- **mensuelle uniquement**
- exécution recommandée : **le 1er jour du mois à 00:05 Europe/Paris**
- le job produit le snapshot du **mois précédent**

**Exemple :**
- exécution le `2026-02-01 00:05 Europe/Paris`
- production du point `as_of_date = 2026-01-31`

**Règle d’idempotence :**
tout job doit fonctionner en **UPSERT** sur la clé naturelle :

- pour `treasury_position_snapshots` : `(tenant, company_id, as_of_date)`
- pour `ar_totals_snapshots` : `(tenant, company_id, as_of_date)`
- pour `ap_totals_snapshots` : `(tenant, company_id, as_of_date)`

**Consigne opérationnelle :**
une relance du job sur une période déjà calculée ne doit **jamais** créer de doublon.

---

### 2.3 Contrat d’exposition (séries)

**Décision :** les endpoints de série sont exposés par le **service Vault**.  
**Linky** agit comme **consommateur** et n’embarque pas de logique métier de recalcul série.

**Motif :**
- cohérence avec l’architecture cible ;
- suppression du risque de divergence entre calculs Vault et calculs Linky ;
- un seul propriétaire du contrat de données métier.

**Endpoints cibles v1 :**
- `GET /ui/aggregations/treasury-series`
- `GET /ui/aggregations/ar-series`
- `GET /ui/aggregations/bfr-series`

**Contrat de réponse :**
aligné sur la spec consolidée §11, avec au minimum :
- `metric`
- `granularity`
- `period`
- `currency`
- `scope`
- `points[]`

Chaque point contient :
- `date`
- `value`
- `secondary` optionnel
- `state` optionnel
- `partial_reason` optionnel

**Convention v1 :**
- `granularity = "month"` uniquement

---

### 2.4 Convention de date

**Décision :** l’`as_of_date` d’un snapshot mensuel correspond au **dernier jour calendaire du mois**.

**Exemples :**
- janvier 2026 → `2026-01-31`
- février 2026 → `2026-02-28`

**Motif :**
- simplicité ;
- stabilité de calcul ;
- meilleure lisibilité fonctionnelle ;
- recette plus claire qu’avec une logique de “dernier jour ouvré”.

**Conséquence API :**
- `points[].date` correspond au **dernier jour calendaire de la période représentée**

---

### 2.5 Choix v1 — granularité

**Décision :** ✅ **granularité mensuelle uniquement pour la v1**

**Consignes :**
- l’API doit l’exprimer explicitement via `granularity: "month"`
- la recette doit couvrir les cas d’affichage sur granularité mensuelle
- la cible long terme reste la granularité journalière, non incluse dans cette ADR

---

### 2.6 Source des données — Vault source exclusive

**Décision :** ✅ **Vault = source exclusive**

Les jobs de snapshotting :
- s’appuient exclusivement sur le **Vault**
- lisent les agrégats métier Vault nécessaires
- alimentent des tables dérivées du Vault
- n’utilisent aucune autre source métier pour les montants affichés

**Tracé retenu :**

**Vault (source)** → **jobs de snapshotting** → **tables de snapshots** → **endpoints de série Vault** → **Linky (affichage)**

Aucune autre base ou microservice n’est autorisé comme source métier pour les blocs Évolution.

---

## 3. Conventions complémentaires

### 3.1 Timezone de référence

La timezone fonctionnelle et technique de référence pour les snapshots mensuels est :

**`Europe/Paris`**

Elle s’applique :
- au déclenchement du job
- à la détermination de la période clôturée
- à l’interprétation de `as_of_date`

---

### 3.2 Politique de relance / correction

Un snapshot mensuel est **révisable par relance contrôlée** du job, tant que l’opération reste idempotente.

Cela autorise :
- le recalcul d’un mois donné ;
- le backfill historique ;
- la correction d’un snapshot erroné sans création de doublon.

La relance doit **remplacer / mettre à jour** le snapshot existant de la période concernée.

---

### 3.3 Données tardives

Si des données Vault nécessaires à un mois donné sont intégrées après la première exécution du job mensuel, le snapshot de ce mois peut être **recalculé** via relance contrôlée.

**Conséquence :**
- en v1, un snapshot mensuel n’est pas juridiquement “figé” ;
- il est considéré comme **reconstruisible** tant que la politique de relance contrôlée est respectée.

---

## 4. Conséquences

### 4.1 Décisions actées

Les points suivants sont désormais verrouillés :
- emplacement des tables ;
- responsabilité des jobs ;
- contrat d’exposition ;
- convention de date ;
- granularité v1 ;
- source exclusive Vault.

### 4.2 Conséquences d’implémentation

Les epics **E2, E3, E4** peuvent démarrer.

**À implémenter :**
1. migrations Vault pour les tables `snapshots.*`
2. jobs mensuels idempotents dans Vault
3. endpoints de série Vault conformes au contrat §11
4. branchement front Linky sur :
   - `TresoreriePositionCard`
   - `EncoursCard`
   - `WorkingCapitalCard`
5. utilisation de `COVERAGE_PARTIAL_THRESHOLD` côté logique de rendu Trésorerie

### 4.3 Risques maîtrisés

Cette ADR réduit :
- le risque de divergence entre Linky et Vault ;
- le risque de duplication de calcul métier ;
- le risque d’implémentations contradictoires entre E2, E3 et E4.

---

## 5. Implémentation

1. Référencer cette ADR dans le plan Scrum (Epic E0)
2. Créer les tables Vault dans le schéma `snapshots`
3. Implémenter les jobs mensuels avec UPSERT
4. Exposer les endpoints de série côté Vault
5. Brancher Linky sur les endpoints Vault
6. Démarrer E2, puis E3, puis E4 selon backlog
